# design-v6 — 触发规则调整与跟单时间门控

> 日期：2026-03-04  
> 前置：design-v1（触发规则定义）、design-v5（最近一轮文档轮次）

---

## 1. 需求概述

本轮包含 2 个独立改动点（仅业务规则调整）：

1. **flag_order 初始化与触发规则（09:30 之后）**：`flag_order` 的初始化条件改为“价格触发”（逐笔成交价 `> base_price * 1.07`）。  
   在 `flag_order` 初始化之前，不做“新增 50 万累计”的触发；初始化之后，后续触发仅由“涨停价卖出档位累计新增达到 50 万”驱动。
2. **trade_return_monitor 跟单门控**：对“对敲跟单”增加条件：两笔回报的 `local_time` 间隔 `<= 5ms` 才允许跟单；若 `> 5ms`，不进行对敲跟单（不发送 100 股涨停价买入委托）。

## 2. 变更 1：flag_order 初始化改为价格触发，后续仅 50 万累计触发

### 2.1 现状（design-v1）

design-v1 的 **1.1 触发规则（09:30 之后）**定义为：满足任一条件触发“挂一笔涨停价卖单”：

1. **价格触发**：逐笔成交价 `price_tick > base_price * 1.07`
2. **首次 50 万触发**：`flag_order` 第一次确认后，第一次累计新增达到 50 万（`trigger_count_50w == 1`）
3. **每次 50 万触发**：每次“涨停价卖出档口新增 50 万”都要再触发一次

### 2.2 新规则（本轮）

本轮同时调整 `flag_order` 的初始化时点，以及“是否允许 50 万累计触发”的门控规则；统计口径与日志口径本轮不改变。

- **flag_order 初始化改为价格触发**：09:30 之后，只有当逐笔成交价首次满足 `price_tick > base_price * 1.07` 时，才初始化 `flag_order`。  
  - `flag_order` 建议取“触发该笔成交的卖方委托号”（卖侧 `askOrder`）作为基准线。
  - 初始化时应将“新增累计金额”清零，从该基准线之后开始统计。
  - 同时将该次“价格触发”作为**首次触发事件**：触发一次下单闭环（挂 100 股涨停价卖单等后续流程按既有定义执行）。
- **50 万累计触发门控**：
  - `flag_order` 未初始化前：不触发任何 50 万累计（即使涨停价卖出档位累计新增达到 50 万也不触发）。
  - `flag_order` 初始化后：后续触发仅由“涨停价卖出档位累计新增达到 50 万”驱动；每次累计新增达到 50 万触发一次（含第一次）。
- **价格触发仅作为初始化起点**：价格触发在本轮仍只需处理一次（首次发生即可），之后不再作为后续触发源。

### 2.3 判定口径（沿用既有定义）

- 生效时间：`>= 09:30:00`（与项目中 After0930 的判定口径保持一致）
- 价格触发：
  - `price_tick`：逐笔成交价（成交流 price）
  - `base_price`：基准价（沿用 design-v1 的定义）
  - 建议仍采用整数比较避免浮点误差（参考 `docs/代码规范.md`）：
    - `tick_raw * 100 > base_raw * 107`

### 2.4 验收点（行为级）

- 场景 A：09:30 后涨停价卖出档位累计新增达到 50 万（甚至多次），但成交价始终不满足 `> base_price * 1.07`：
  - 预期：`flag_order` 不初始化；不触发任何 50 万累计相关的下单闭环。
- 场景 B：首次发生价格触发（`> base_price * 1.07`）：
  - 预期：初始化 `flag_order`（以该笔成交的卖方委托号为基准）并清零累计，同时触发一次首次下单闭环；
  - 随后仅当涨停价卖出档位累计新增达到 50 万时才触发下单闭环（每 50 万一次，含第一次）。

## 3. 变更 2：trade_return_monitor 跟单增加 5ms 时间门控

### 3.1 现状（trade_return_monitor.cpp）

当前 `TradeReturnMonitor::on_match()` 的“对敲跟单”动作发生在 `reason == same_price_second_sale` 路径：

- 触发前置（摘要）：
  - 本机时间 `>= 09:30:00`
  - `snapshot_ready_ == true`（09:17 快照完成后）
  - 仅 SZ 市场，且为卖出回报（`EntrustType == SALE`）
  - 命中 `same_price_second_sale`：第二笔卖出回报的 `OrderPrice` 与记录价格相同（`fabs(msg.OrderPrice - recorded.order_price) < 1e-6`）
- 跟单动作（摘要）：
  - 发送 100 股限价买入委托（价格使用记录价）
  - 同一逻辑中还会发送一笔 100 股“本方最优价格”买入委托（同 stock，同价格参数）

### 3.2 新规则（本轮）

在执行跟单动作前新增时间门控条件：

- 构成跟单判定的两笔回报，其 `local_time` 间隔必须 `<= 5ms`
- 若 `local_time` 间隔 `> 5ms`：
  - 不进行对敲跟单：不发送 100 股涨停价买入委托（以及同一跟单动作内的其他买入委托）
  - 建议打印一条明确的 skip 日志（包含 `delta_ms`），方便线下核对

### 3.3 local_time 口径说明

- `local_time` 指回报到达时记录的**本机时间**（当前日志中以 `local_time=YYYY-MM-DD HH:MM:SS.mmm` 形式打印）。
- 实现建议（仅文档建议）：
  - 时间差计算应基于时间点/毫秒计数（如 ms since epoch 或 `steady_clock`），避免使用字符串做差；
  - 需要为“第一笔回报”在 per-stock 维度保存本地时间戳，第二笔到达时计算 `delta_ms` 并据此门控。

### 3.4 验收点（行为级）

- 场景 A：两笔回报到达的 `local_time` 间隔 `<= 5ms`：
  - 预期：保持现有行为，触发跟单下单；日志中出现 `FOLLOWUP` 相关输出。
- 场景 B：两笔回报到达的 `local_time` 间隔 `> 5ms`：
  - 预期：不发送跟单买入委托；日志清晰记录 skip 原因与 `delta_ms`。
